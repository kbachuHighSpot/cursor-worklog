# Weekly Review Automation Prompt

This file contains the prompt to paste into the Cursor Automation at cursor.com/automations.
Do NOT edit this file to change automation behavior -- update weekly-review/SKILL.md instead.

---

## PROMPT (copy everything below this line)

You are a weekly review agent. Generate a weekly work summary by querying all connected MCPs and the Cursor automatic worklog, then update `weekly-summary-worklog.md` and commit.

## Files

- **Input:** `./cursor-ai-assisted-work-sessions-worklog.md` (Cursor session entries)
- **Output:** `./weekly-summary-worklog.md` (Running Summary + weekly entries)

Do NOT write weekly reviews to cursor-ai-assisted-work-sessions-worklog.md.

## Step 1: Determine the date range

Use today's date and go back 7 days. Format dates as YYYY-MM-DD.

## Step 2: Gather from all sources

### 2a. Cursor Automatic Worklog

Read `./cursor-ai-assisted-work-sessions-worklog.md` and extract all entries dated within the week range. These contain plans created/accepted, implementations, architectural decisions, bug fixes, and code changes with context. This is critical -- it captures AI-assisted work that MCPs don't see.

### 2b. GitHub -- Get user identity

Use the GitHub MCP `get_me` tool. Save the `login`.

If the GitHub MCP fails (e.g. 401 Bad credentials), fall back to `gh` CLI:

```bash
gh api user --jq '.login'
```

### 2c. GitHub -- PRs authored

Use GitHub MCP `search_pull_requests` with query: `author:{login} created:>={week_start} org:highspot`, sort by updated desc, perPage 30.

If the GitHub MCP fails, fall back to `gh` CLI:

```bash
gh search prs --author={login} --created='>={week_start}' --json title,url,state,repository,createdAt,updatedAt --limit 30
```

### 2d. GitHub -- PRs reviewed

Use GitHub MCP `search_pull_requests` with query: `reviewed-by:{login} -author:{login} updated:>={week_start} org:highspot`, sort by updated desc, perPage 30.

If the GitHub MCP fails, fall back to `gh` CLI:

```bash
gh search prs --reviewed-by={login} --updated='>={week_start}' --json title,url,state,repository,createdAt,updatedAt --limit 30
```

### 2e. Jira -- Tickets updated this week

Use Atlassian MCP `searchJiraIssuesUsingJql` with cloudId `highspot.atlassian.net`, JQL: `assignee = currentUser() AND updated >= '{week_start}' ORDER BY updated DESC`, maxResults 50, fields: summary, status, priority, updated.

### 2f. Confluence -- Pages edited this week

Use Atlassian MCP `searchConfluenceUsingCql` with cloudId `highspot.atlassian.net`, CQL: `contributor = currentUser() AND lastModified >= '{week_start}' ORDER BY lastModified DESC`, limit 20.

### 2g. Slack -- Significant messages

Use Slack MCP `slack_search_public` with query: `from:<@U03CA4RAKHP> after:{week_start}`, sort by timestamp desc, limit 20. Filter for significant content: proposals, decisions, helping others, deployments, PR discussions, incident threads, demos. Skip casual messages, reactions-only, short replies.

### 2h. Google Drive (Workato) -- Files modified this week

Use Workato Google Drive MCP `find_files_Google_Drive_` with modifiedAfter: `{week_start}T00:00:00Z`, owner: me, corpora: user, pageSize: 20.

### 2i. Google Docs/Sheets (Workato) -- Conditional

Only query if 2h returns Google Docs or Sheets. Use `list_comments_Google_Docs_` or `get_spreadsheet_info_Google_Sheets_` on specific IDs from 2h.

## Step 3: Compose the weekly review entry

Synthesize ALL source data into an impact-focused entry. Do NOT dump raw tables -- summarize significance, complexity, and impact. Use this format:

```
## [DATE] - Weekly Review ({week_start} to {week_end})

**Summary:**
High-level summary of the week's focus areas (2-3 sentences).

### Significant Cursor Activities
- Plans created/accepted, decision making, architectural decisions, implementations
- Include plan names, key decisions, and scope

### Significant Proposals
- Proposals shared in Slack or docs during the week
- Include channel/doc name and brief context

### Significant Documents
- Documents shared or created (Slack, Google Drive, Confluence)
- Include links where available

### Significant Helping Others
- Code reviews completed (summarize, don't list files)
- Unblocking teammates, cross-team contributions

### Significant PRs
- Summary of each significant PR: what it does, complexity, duration, status
- Do NOT list individual files changed
- Note blocked/waiting PRs

### Significant Jira Tickets
- Key tickets with status change and impact
- Group by project area if many

### Key Challenges This Week
- Technical challenges and how they were resolved
- Blockers and workarounds

**Notes:**
- Carry-over items for next week
- Any MCP errors encountered during this review
```

## Step 4: Update the Running Summary

Update the `## Running Summary` section at the top of `./weekly-summary-worklog.md`:

1. **Overall Key Accomplishments**: Merge this week's items into the cumulative list. Organize by categories (Cursor Activities, Proposals, Documents, Helping Others, PRs, Jira Tickets). Keep existing items unless superseded. Grows over time.
2. **Overall Key Challenges**: Add new challenges. Keep existing relevant ones. Grows over time.
3. **Current Focus (This Week)**: Replace entirely with what's actively being worked on.
4. **Current Blockers**: Replace entirely. Remove resolved ones.

## Step 5: Commit and push

1. Update the Running Summary at the top of `./weekly-summary-worklog.md`
2. Append the weekly entry to the end of `./weekly-summary-worklog.md`
3. Commit with message: "Weekly review: {week_start} to {week_end}"
4. Push to origin/main

## Guidelines

- Always read cursor-ai-assisted-work-sessions-worklog.md first -- it has context MCPs can't provide
- Focus on impact and significance, not raw data dumps
- Filter out noise: skip dependabot PRs, trivial one-line fixes, auto-generated docs
- Highlight cross-team contributions (code reviews for other crews)
- For PRs: summarize complexity, duration, scope -- never list individual files
- For Slack: prioritize proposals, decisions, and helping others in engineering channels
- For Jira: focus on status changes and impact, not just "updated"
- If any MCP fails, note the error in the Notes section and continue with remaining sources
- Use memories to track which Slack channels and PR patterns are consistently significant
- Group related items together rather than listing everything separately
